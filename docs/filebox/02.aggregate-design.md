# FileBox Aggregate 작업계획

## 목적

이 문서는 `docs/filebox/01-spec-design.md`와 `docs/filebox/01-spec-design-implement.md`를 기준으로 `slcn-aggregate` 모듈을 FileBox
설계에 맞게 전환하기 위한 작업계획서다.

Spec 단계에서는 Travel/Trip 공개 계약이 이미 FileBox 중심으로 변경되었다. 그 결과 현재 `slcn-aggregate`는 컴파일되지 않는다. Aggregate 단계의 목표는 기존
Travel/Trip 저장 및 조회 흐름을 새 Spec 계약에 맞게 바꾸고, 다음 단계인 `slcn-rest` 전환이 가능하도록 business/store 레이어를 안정화하는 것이다.

## 현재 상태 요약

### Spec 변경으로 인한 현재 컴파일 실패

`./gradlew :slcn-aggregate:compileJava`는 현재 실패한다.

주요 원인:

- `TravelLogic`, `TravelStore`, `TravelJpoMapper`가 삭제된 `com.seoulchonnom.spec.travel.entity.TravelDay`, `TravelPlace`,
  `TravelReview`, `TravelTag`, `TravelPhoto`를 import한다.
- `TravelLogic`이 삭제된 `TravelPhotoCdo`를 사용한다.
- `TravelMapper` 호출 시그니처가 `Travel + List<FileBoxItemRdo>`로 바뀌었지만 aggregate는 아직 하위 엔티티 목록을 넘긴다.
- `TripLogic`이 삭제된 `TripCdo.logoFileId`, `firstMapFileId`, `secondMapFileId`와 `Trip`의 직접 파일 ID getter를 사용한다.
- `TripJpo`는 아직 `logo_file_id`, `first_map_file_id`, `second_map_file_id` 컬럼을 필수 필드로 가진다.

### 현재 Travel 저장 구조

현재 aggregate는 Travel root와 하위 객체를 별도 JPO/Repository로 저장한다.

- `TravelJpo`
- `TravelDayJpo`
- `TravlPlaceJpo`
- `TravelPhotoJpo`
- `TravelTagJpo`
- `TravelReviewJpo`

현재 로직 특징:

- Travel 생성 시 기간 기준으로 `TravelDay` 엔티티를 자동 생성한다.
- Travel 수정 시 기간 축소 대상 day의 content/place/photo 존재 여부를 확인한다.
- day/place/photo/tag/review는 모두 별도 store save/delete 경로를 가진다.
- 사진은 `TravelPhotoJpo`로 저장되고 `travelDayId`, `travelPlaceId`는 기존 하위 엔티티 ID를 참조한다.

### 현재 Trip 저장 구조

현재 Trip은 파일 참조를 root JPO 필드로 직접 가진다.

- `logoFileId`
- `firstMapFileId`
- `secondMapFileId`

현재 로직 특징:

- Trip 생성 시 `TripCdo`의 직접 파일 ID를 검증한다.
- `FileAssetStore.findById`로 파일 타입을 검증한다.
- Trip 조회 시 직접 파일 ID를 다시 `FileAssetRdo`로 조립한다.

## Aggregate 설계 방향

### FileBox 저장 방식

FileBox 엔티티는 PostgreSQL JPA가 아니라 MongoDB에 저장한다.

이유:

- `FileAsset`도 현재 MongoDB에 저장되고 있어 파일 메타데이터와 파일 연결 정보의 저장소 성격이 맞다.
- `FileBoxItem`은 owner 내부 VO 목록이고, item 단독 API/수정 생명주기가 없으므로 document 저장 방식과 잘 맞다.
- Travel/Trip 조회 시 owner 기준으로 FileBox 전체를 한 번에 가져오는 사용 패턴이다.
- 추후 FileBox item 수가 늘어나도 Mongo document 구조에서 owner 단위 조회를 단순하게 유지할 수 있다.

Aggregate에 FileBox Mongo store 패키지를 추가한다.

```text
com.seoulchonnom.aggregate.filebox
└── store
    ├── FileBoxStore.java
    ├── doc
    │   └── FileBoxDoc.java
    ├── mapper
    │   └── FileBoxDocMapper.java
    └── repository
        └── FileBoxRepository.java
```

저장 모델:

```java

@Document(collection = "file_box")
@CompoundIndex(name = "uk_file_box_owner", def = "{'ownerType': 1, 'ownerId': 1}", unique = true)
public class FileBoxDoc {
	@Id
	private String id;
	@Indexed
	private FileBoxOwnerType ownerType;
	@Indexed
	private String ownerId;
	private List<FileBoxItem> items;
	private long registeredTime;
	private long modifiedTime;
}
```

Repository:

```java

@Repository
public interface FileBoxRepository extends MongoRepository<FileBoxDoc, String> {
	Optional<FileBoxDoc> findByOwnerTypeAndOwnerId(FileBoxOwnerType ownerType, String ownerId);

	List<FileBoxDoc> findAllByOwnerTypeAndOwnerIdIn(FileBoxOwnerType ownerType, Collection<String> ownerIds);

	void deleteByOwnerTypeAndOwnerId(FileBoxOwnerType ownerType, String ownerId);
}
```

판단:

- `FileBoxItem`은 Spec에서 VO이므로 별도 Mongo collection보다 `FileBoxDoc.items` embedded list로 시작한다.
- 한 owner는 하나의 FileBox만 가진다는 설계와 잘 맞는다.
- item 단위 단독 조회/수정 API가 없으므로 item repository를 만들지 않는다.
- MongoDB unique compound index는 `ownerType + ownerId`에 둔다.
- 추후 item 단위 조건 검색이나 페이징이 필요하면 embedded list 유지 여부를 별도 단계로 검토한다.

### FileBoxStore 책임

`FileBoxStore`는 FileBox aggregate 저장/조회/삭제만 담당한다.

필요 메서드:

```java
FileBox save(FileBox fileBox);

FileBox findByOwner(FileBoxOwnerType ownerType, String ownerId);

Optional<FileBox> findOptionalByOwner(FileBoxOwnerType ownerType, String ownerId);

FileBox createForOwner(FileBoxOwnerType ownerType, String ownerId);

FileBox syncItems(FileBoxOwnerType ownerType, String ownerId, List<FileBoxItem> items);

void deleteByOwner(FileBoxOwnerType ownerType, String ownerId);
```

정책:

- Travel/Trip 생성 시 Logic에서 `createForOwner` 또는 `syncItems`를 호출해 자동 생성한다.
- FileBox 자체를 외부 API로 만들지 않는다.
- 삭제는 owner 삭제 흐름에서 함께 수행한다.
- 신규 item id는 Logic 또는 FileBoxStore의 sync 직전에 `UUID.randomUUID().toString()`으로 생성한다.
- DB 자동 생성 id를 `FileBoxItem.id`로 쓰지 않는다.

## Travel Aggregate 전환 계획

### Travel JPO 정리

`TravelJpo`에서 제거할 필드:

- `coverPhotoId`
- `oneLineReview`

`TravelJpo`에 추가할 필드:

- `List<TravelDay> days`
- `List<String> tags`
- `TravelReview review`

권장 저장 방식:

- `Travel.days`, `Travel.tags`, `Travel.review`는 Travel root의 내부 VO이므로 JSON 컬럼으로 저장한다.
- 기존 `TravelDayJpo`, `TravelPlaceJpo`, `TravelTagJpo`, `TravelReviewJpo`는 aggregate 전환 후 사용하지 않는다.
- 기존 `TravelPhotoJpo`는 FileBox로 대체하므로 사용하지 않는다.

필요 converter:

```text
TravelDayListConverter
TravelTagListConverter
TravelReviewConverter
```

주의:

- `ddl-auto=update`는 기존 테이블/컬럼을 자동 삭제하지 않는다.
- 개발 중 프로젝트이므로 기존 JPO 클래스를 제거해도 DB 잔여 테이블은 남을 수 있다.
- 잔여 테이블 정리는 별도 SQL 또는 수동 DB 초기화 정책으로 처리한다.

### TravelStore 정리

`TravelStore`는 Travel root 저장/조회/삭제 중심으로 축소한다.

유지:

- `save(Travel travel)`
- `findById(String travelId)`
- `findAllVisible()`
- `deleteTravel(Travel travel)`

제거:

- day/place/photo/tag/review 별도 save/find/delete 메서드
- `existsPlaceByDayIds`
- `existsPhotoByDayIds`
- `findTagsByTravelId`
- `findReviewByTravelId`
- `findPhotosByTravelId`
- `findPlacesByTravelId`
- `findDaysByTravelId`

삭제 정책:

- `deleteTravel`은 Travel root 삭제만 담당하지 않고 Logic에서 FileBox 삭제를 함께 호출한다.
- Store가 다른 aggregate인 FileBoxStore를 직접 의존하지 않도록 한다.

### TravelLogic 생성 흐름

변경 후 생성 흐름:

1. `TravelCdo` 기본 필드 검증
2. 기간 검증
3. `TRAVEL-{번호}` ID 생성
4. 기간 기준 `TravelDay` VO 목록 생성
5. 요청의 `travelDays`가 있으면 날짜 기준으로 day VO에 title/memo/place를 반영
6. tags 검증 후 `List<String>`으로 Travel root에 저장
7. review가 있으면 `TravelReview` VO로 저장
8. Travel root 저장
9. FileBox owner 생성 및 `files` 동기화
10. 저장된 Travel과 FileBox item을 조립해 `TravelDetailRdo` 반환

Travel 생성 시 FileBox는 항상 생성한다.

- `files`가 null 또는 empty여도 빈 FileBox를 생성한다.
- 단, root cover 필수 정책을 유지할지 결정해야 한다.
- 기존 Travel은 `coverPhotoId` 필수였다. 새 설계에서는 `files` 안에 `targetType=TRAVEL`, `role=COVER` item이 필수인 것으로 해석한다.

권장 검증:

- Travel root `COVER`는 필수 1개
- Travel root `GALLERY`는 0개 이상
- Travel day/place `COVER`는 해당 day/place에 content가 있거나 FE가 보낸 경우 선택 0 또는 1개
- Travel day/place `GALLERY`는 0개 이상
- Travel FileBox item의 `fileAssetId`는 모두 `FileType.TRAVEL`이어야 한다.

### TravelLogic 수정 흐름

변경 후 수정 흐름:

1. 기존 Travel 조회
2. 기간 검증
3. target 기간 날짜 목록 생성
4. 기존 Travel.days에서 삭제될 날짜 산출
5. 삭제될 날짜에 content/place/file item이 있으면 `confirmDeleteDays` 확인
6. 유지 날짜는 기존 VO를 날짜 기준으로 유지하고 dayNumber/sortOrder 갱신
7. 신규 날짜는 새 `TravelDay` VO 생성
8. 요청 `travelDays`가 있으면 날짜 기준으로 day/place 정보 동기화
9. tags/review 동기화
10. Travel root 저장
11. FileBox item 목록 동기화
12. 삭제된 날짜에 해당하는 `targetType=TRAVEL_DAY` item 삭제
13. 삭제된 placeKey에 해당하는 `targetType=TRAVEL_PLACE` item 삭제
14. 조립 응답 반환

날짜 식별 정책:

- `TravelDay.date`가 day 식별자다.
- 기존 날짜 변경은 지원하지 않는다.
- 요청 day의 date가 Travel 기간 밖이면 `BadRequestException`.
- 요청 day의 date가 중복되면 `BadRequestException`.

placeKey 정책:

- `TravelPlace.placeKey`는 FE가 생성한 UUID 문자열이다.
- 신규 place에 `placeKey`가 없으면 `BadRequestException`.
- UUID 형식이 아니면 `BadRequestException`.
- 같은 Travel 안에서 중복되면 `BadRequestException`.
- 기존 placeKey 변경은 삭제+신규 생성으로 처리한다.

기간 축소 conflict 판단:

삭제될 날짜에 아래 중 하나라도 있으면 content가 있다고 본다.

- day title
- day memo
- day places
- FileBox item 중 `targetType=TRAVEL_DAY`, `targetId={date}`
- FileBox item 중 삭제될 날짜의 placeKey를 targetId로 가진 `targetType=TRAVEL_PLACE`

`confirmDeleteDays=true`이면 해당 날짜/장소 target FileBox item도 함께 삭제한다.

### Travel 응답 조립

`TravelMapper`는 이미 `Travel + List<FileBoxItemRdo>`를 받는다.

Aggregate의 역할:

1. `FileBoxStore.findByOwner(TRAVEL, travelId)`로 FileBox 조회
2. FileBox item의 `fileAssetId` 목록으로 `FileAssetStore.findById` 조회
3. `FileBoxMapper.toFileBoxItemRdo(item, FileAssetRdo.from(fileAsset))`로 item RDO 조립
4. `TravelMapper.toTravelDetailRdo(travel, fileItemRdos)` 호출

목록 조회 `getTravels()`도 root cover가 필요하다.

- 각 Travel별 FileBox를 조회한다.
- `targetType=TRAVEL`, `role=COVER` item의 FileAsset을 붙여 `FileBoxItemRdo`를 만든다.
- `TravelMapper.toTravelRdo(travel, cover)` 호출

초기 구현에서는 N+1 조회를 허용할 수 있다. 목록 성능이 문제가 되면 ownerId 목록 기반 FileBox bulk 조회와 FileAsset bulk 조회를 추가한다.

## Trip Aggregate 전환 계획

### Trip JPO 정리

`TripJpo`에서 제거할 필드:

- `logoFileId`
- `firstMapFileId`
- `secondMapFileId`

유지:

- `date`
- `type`
- `name`
- `nextButtonText`
- `previousButtonText`
- `driveUrl`
- `quiz`

`TripJpoMapper`는 Spec `Trip`과 JPO의 동일 필드만 매핑하도록 유지한다.

### TripLogic 생성 흐름

변경 후 생성 흐름:

1. quiz 검증
2. files 검증
3. navigation 필드 검증
4. `TRIP-{번호}` ID 생성
5. `Trip` 저장
6. FileBox owner 생성 및 files 동기화
7. FileBox role item을 FileAssetRdo로 조립해 `TripDetailRdo` 반환

Trip file role 검증:

- `targetType`은 모두 `TRIP`
- `targetId`는 모두 null
- `LOGO` 필수 1개, `FileType.LOGO`
- `FIRST_MAP` 필수 1개, `FileType.MAP`
- `SECOND_MAP` 선택 0 또는 1개, 있으면 `FileType.MAP`
- `GALLERY`, `COVER`는 Trip에서 허용하지 않는다.
- 같은 role 중복은 `BadRequestException` 또는 `InvalidTripRegisterException`으로 처리한다.

현재 Trip 로직은 `InvalidTripRegisterException`을 사용한다. Trip 도메인의 기존 예외 일관성을 위해 Trip files 검증 실패도
`InvalidTripRegisterException`으로 통일한다.

Navigation 정책:

- `SECOND_MAP`, `nextButtonText`, `previousButtonText`는 모두 있거나 모두 없어야 한다.
- 기존 정책을 유지하되 `SECOND_MAP` 존재 여부는 `files`의 role로 판단한다.

### Trip 조회 조립

Trip 응답은 FileBox item 메타데이터를 노출하지 않는다.

Aggregate의 역할:

1. `TripStore.findById(tripId)`로 Trip 조회
2. `FileBoxStore.findByOwner(TRIP, tripId)` 조회
3. role별 FileBox item 추출
4. `FileAssetStore.findById`로 FileAsset 조회
5. `TripMapper.toTripDetailRdo(trip, logo, firstMap, secondMap)` 호출

목록 조회:

- 각 Trip별 FileBox에서 `LOGO` role item을 찾는다.
- `TripMapper.toTripListRdo(trip, logo)` 호출

### Trip 수정 API 고려

Spec에는 `TripUdo`가 추가되었지만 현재 `TripFacade`와 `TripResource`에는 수정 API가 없다.

Aggregate 단계에서는 선택지가 있다.

권장:

- 이번 Aggregate 단계에서는 public `modifyTrip` 로직까지 구현하지 않는다.
- 단, 내부 helper는 `TripCdo`와 `TripUdo`가 같은 files 검증/동기화 정책을 재사용할 수 있게 분리한다.
- Rest 단계에서 Trip 수정 API를 추가할 때 `TripLogic.modifyTrip`을 구현한다.

이유:

- 현재 REST 계약이 없는 기능을 aggregate에 먼저 public으로 추가하면 테스트/사용 경로가 불명확하다.
- 이번 단계의 우선순위는 Spec 변경으로 깨진 기존 생성/조회 흐름 복구다.

## FileBox item 검증 공통 정책

공통 검증:

- `fileAssetId` 필수
- `targetType` 필수
- `role` 필수
- `targetType=TRAVEL_DAY`이면 `targetId` 필수, `yyyy-MM-dd` 형식
- `targetType=TRAVEL_PLACE`이면 `targetId` 필수, UUID 형식
- `targetType=TRAVEL` 또는 `TRIP`이면 `targetId == null`
- 같은 owner 안에서 `targetType + targetId + role + fileAssetId` 중복 금지
- `sortOrder == null`이면 같은 `targetType + targetId + role` 그룹 내 순서대로 부여

수정 검증:

- `FileBoxItemUdo.id`가 있으면 기존 owner의 item이어야 한다.
- `FileBoxItemUdo.id`가 없으면 신규 item으로 보고 애플리케이션에서 UUID를 생성한다.
- FE가 보낸 신규 item id는 허용하지 않는다.
- 기존 item id가 다른 owner에 속하면 거절한다.

구현 위치:

- FileBox item id 부여와 item 목록 동기화는 `FileBoxStore` 또는 별도 `FileBoxItemSyncer` helper로 둔다.
- Travel/Trip별 target/role 정책 검증은 각 Logic에 둔다.
- 파일 타입 검증은 각 Logic에서 `FileAssetStore`를 통해 수행한다.

## 구현 순서

### 1단계: FileBox store 추가

1. `FileBoxDoc` 추가
2. `FileBoxRepository extends MongoRepository<FileBoxDoc, String>` 추가
3. `FileBoxDocMapper` 추가
4. `FileBoxStore` 추가
5. `FileBoxStoreTest` 또는 mapper 테스트 추가

완료 기준:

- FileBox 저장/조회/삭제 단위 테스트 통과
- Spec `FileBox`와 Mongo `FileBoxDoc` 간 변환 통과
- `ownerType + ownerId` 기준 단건 조회와 중복 방지 정책 확인

### 2단계: Trip aggregate 전환

Trip은 Travel보다 범위가 작으므로 먼저 전환한다.

1. `TripJpo` 직접 파일 ID 필드 제거
2. `TripLogic` 생성/조회에서 FileBoxStore 사용
3. Trip files role 검증 구현
4. Trip 응답 조립을 FileBox role 기반으로 변경
5. `TripLogicTest`, `TripJpoMapperTest` 수정

완료 기준:

- `:slcn-aggregate:compileJava`의 Trip 관련 오류 제거
- Trip aggregate 테스트 통과

### 3단계: Travel aggregate 전환

1. `TravelJpo`를 root + JSON VO 저장 구조로 변경
2. 기존 Travel 하위 JPO/repository/store 메서드 제거
3. `TravelJpoMapper`를 root 매핑 중심으로 단순화
4. `TravelLogic`을 Travel root 내부 VO 동기화 방식으로 재작성
5. Travel FileBox item 검증 구현
6. Travel 상세/목록 응답 조립 변경
7. 기간 축소 conflict 판단을 날짜/placeKey/FileBox item 기준으로 변경
8. `TravelLogicTest`, `TravelStoreTest` 수정

완료 기준:

## 테스트 계획

- owner별 단건 조회

- owner 삭제 시 FileBox 삭제
- item id 보존/생성 정책
- `ownerType + ownerId` 중복 생성 방지

- Travel 삭제 시 FileBox 삭제

## 위험 요소와 결정 사항

### 기존 데이터/테이블

개발 중 프로젝트이고 수정 영향도는 감수 가능한 상태다. 따라서 구현은 새 설계 기준으로 단순하게 간다.

다만 PostgreSQL의 `ddl-auto=update`는 삭제된 컬럼/테이블을 자동 제거하지 않는다.

남을 수 있는 DB 구조:

- `travel_days`
- `travel_places`
- `travel_photos`
- `travel_tags`
- `travel_reviews`
- `trip.logo_file_id`
- `trip.first_map_file_id`
- `trip.second_map_file_id`

후속 대응:

- 개발 DB는 초기화 또는 수동 DDL로 정리한다.
- 코드에서는 더 이상 해당 JPO/repository를 사용하지 않는다.
- FileBox는 MongoDB `file_box` collection에 신규 저장한다.

### JSON/Document 저장 방식

이번 계획은 저장 대상을 분리한다.

- Travel 내부 VO(`days`, `tags`, `review`)는 PostgreSQL `TravelJpo`의 JSON 컬럼으로 저장한다.
- FileBox는 MongoDB `file_box` document로 저장하고, `items`는 document 내부 embedded list로 저장한다.

장점:

- Spec 설계의 VO 생명주기와 잘 맞는다.
- item 단독 API가 없으므로 구현이 단순하다.
- Travel 전체 수정/조회 흐름과 맞다.
- FileAsset과 FileBox가 모두 MongoDB에 있어 파일 관련 저장소 성격이 일관된다.

주의:

- SQL 레벨에서 특정 placeKey/photo 검색이 필요하면 Travel 내부 VO JSON 컬럼은 불리하다.
- MongoDB에서 FileBox item 단위 조건 검색은 가능하지만, embedded list가 매우 커지면 document 크기와 조회 성능을 다시 검토해야 한다.
- 사진 수가 매우 많아지면 FileBox item 별도 collection 분리가 필요할 수 있다.

현재 요구사항에서는 페이징 없이 owner 단위 전체 응답이므로 FileBox embedded document 저장을 선택한다.

### PostgreSQL/MongoDB 일관성

Travel/Trip root는 PostgreSQL에 저장하고 FileBox는 MongoDB에 저장한다. 두 저장소를 하나의 로컬 트랜잭션으로 묶는 구조는 현재 프로젝트에 없다.

구현 기준:

- 생성 흐름은 root 저장 후 FileBox 저장 순서로 처리한다.
- FileBox 저장이 실패하면 전체 요청은 실패로 응답한다.
- 개발 중 프로젝트이므로 초기 구현에서는 분산 트랜잭션을 도입하지 않는다.
- 실패 후 PostgreSQL root만 남는 상황을 줄이기 위해 Logic에서 예외 발생 시 보상 삭제를 검토한다.
- 조회 시 root는 있는데 FileBox가 없으면 빈 FileBox를 생성할지, `FileBoxNotFoundException`으로 실패할지 정책을 정해야 한다.

권장:

- 생성/수정 중 FileBox 저장 실패는 예외를 그대로 전파한다.
- 생성 직후 FileBox 저장 실패 시에는 catch 블록에서 방금 저장한 Travel/Trip root를 삭제하는 보상 처리를 우선 검토한다.
- 조회 시 FileBox가 없는 기존/깨진 데이터는 `findOptionalByOwner` 후 빈 FileBox 자동 생성으로 복구하는 방식을 고려한다. 단, 이 자동 복구는 개발 중 데이터 보정 목적이며
  명시적으로 테스트한다.

### 예외 타입

- Travel 검증 실패는 기존처럼 `BadRequestException`을 사용한다.
- Travel 기간 축소 conflict는 기존처럼 `TravelPeriodConflictException`을 사용한다.
- Trip 등록 검증 실패는 기존처럼 `InvalidTripRegisterException`을 사용한다.
- FileBox owner 조회 실패는 신규 예외를 만들거나 `BadRequestException`을 사용할 수 있다.

권장:

- owner가 반드시 있어야 하는 조회에서 없으면 `BadRequestException("FileBox를 찾을 수 없습니다.")`보다 도메인 전용 `FileBoxNotFoundException`을 추가한다.
- 다만 외부 API로 FileBox가 노출되지 않으므로 처음에는 aggregate 내부 예외로 제한한다.
