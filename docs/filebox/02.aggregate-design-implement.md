# FileBox Aggregate 구현 결과

## 작업 범위

`docs/filebox/02.aggregate-design.md` 기준으로 `slcn-aggregate` 모듈을 FileBox 중심 구조로 전환했다.

이번 단계의 목표는 Spec 변경으로 깨진 Aggregate 컴파일을 복구하고, Travel/Trip business/store 레이어가 새 FileBox 계약을 사용하도록 만드는 것이다. REST 모듈 전환은 다음
단계로 남겼다.

## 서브에이전트 수행 결과

### 코드 작성

코드 작성 서브에이전트를 실행했으나 완료 응답 전에 메인 작업에서 구현과 검증을 마쳤다. 최종 반영 내용은 현재 작업트리 기준이다.

### 정적 검증

정적 검증 서브에이전트는 초기 상태 기준으로 old contract 잔존, Trip 직접 파일 ID 의존, Travel 하위 엔티티 저장 구조, 테스트 미전환 문제를 지적했다.

해당 지적 중 아래 항목은 이번 구현에서 해소했다.

- `slcn-aggregate` production 컴파일 실패
- `TravelPhoto`, `TravelTag`, `coverPhotoId`, Trip 직접 파일 ID production 참조
- FileBox Mongo store와 Travel/Trip Logic 미연결
- Travel root JSON VO 저장 구조 부재
- old contract 기반 aggregate 테스트

## 구현 내용

### FileBox Mongo Store 추가

추가 패키지:

```text
com.seoulchonnom.aggregate.filebox
├── store/FileBoxStore.java
├── store/doc/FileBoxDoc.java
├── store/mapper/FileBoxDocMapper.java
└── store/repository/FileBoxRepository.java
```

구현 기준:

- FileBox는 MongoDB `file_box` collection에 저장한다.
- `FileBoxDoc`는 `ownerType + ownerId` compound unique index를 가진다.
- `FileBoxItem`은 별도 collection이 아니라 `FileBoxDoc.items` embedded list로 저장한다.
- `FileBoxStore.syncItems`는 신규 item id를 애플리케이션에서 UUID로 생성한다.
- `FileBoxItemUdo.id`처럼 기존 id가 들어온 경우 기존 owner item id인지 검증한다.
- owner 삭제 시 `deleteByOwner(ownerType, ownerId)`로 FileBox를 함께 삭제할 수 있게 했다.

### Travel Aggregate 전환

저장 구조:

- `TravelJpo`에서 `coverPhotoId`, `oneLineReview`를 제거했다.
- `TravelJpo`에 `days`, `tags`, `review`를 JSON 컬럼으로 추가했다.
- `TravelDayListConverter`, `StringListConverter`, `TravelReviewConverter`를 추가했다.
- 기존 Travel 하위 JPO/Repository를 제거했다.
    - `TravelDayJpo`
    - `TravelPlaceJpo`
    - `TravelPhotoJpo`
    - `TravelTagJpo`
    - `TravelReviewJpo`
    - 각 하위 repository

Logic 변경:

- Travel 생성 시 기간 기준 `TravelDay` VO를 root 내부에 생성한다.
- `TravelDayUdo.date` 기준으로 day payload를 반영한다.
- `TravelPlaceUdo.placeKey`는 UUID 형식과 중복을 검증한다.
- tags는 `List<String>`으로 root에 저장하고 중복/최대 10개를 검증한다.
- review는 `TravelReview` VO로 root에 저장한다.
- Travel 파일은 `FileBoxStore.syncItems(TRAVEL, travelId, items)`로 저장한다.
- Travel 상세/목록 응답은 `FileBoxStore -> FileAssetStore -> FileBoxMapper -> TravelMapper` 순서로 조립한다.
- 기간 축소 시 삭제될 날짜에 내용 또는 FileBox item이 있으면 `confirmDeleteDays`를 요구한다.
- Travel 삭제 시 FileBox를 함께 삭제한다.

검증 정책:

- Travel root `COVER`는 정확히 1개여야 한다.
- Travel 파일은 `FileType.TRAVEL`이어야 한다.
- Travel 파일 role은 `COVER`, `GALLERY`만 허용한다.
- `TRAVEL` target은 `targetId == null`이어야 한다.
- `TRAVEL_DAY` target은 여행 기간 내 `yyyy-MM-dd` 날짜여야 한다.
- `TRAVEL_PLACE` target은 존재하는 `placeKey` UUID여야 한다.
- 같은 `targetType + targetId + role + fileAssetId` 중복은 거절한다.
- `sortOrder <= 0`이면 같은 target/role 그룹 내 순서대로 부여한다.

### Trip Aggregate 전환

저장 구조:

- `TripJpo`에서 `logoFileId`, `firstMapFileId`, `secondMapFileId`를 제거했다.
- Trip root에는 더 이상 실 파일 ID를 저장하지 않는다.
- Trip 파일 연결은 FileBox item role로만 저장한다.

Logic 변경:

- Trip 생성 시 `TripCdo.files`를 `FileBoxItem`으로 변환한다.
- `LOGO`, `FIRST_MAP`, `SECOND_MAP` role 정책을 검증한다.
- Trip 파일은 `FileBoxStore.syncItems(TRIP, tripId, items)`로 저장한다.
- Trip 상세/목록 응답은 FileBox role item을 `FileAssetRdo`로 조립해 기존 `TripMapper`에 전달한다.
- Trip 응답에는 FileBox item id/role/targetType 같은 연결 메타데이터를 노출하지 않는다.

검증 정책:

- Trip 파일 targetType은 모두 `TRIP`이어야 한다.
- Trip 파일 targetId는 모두 null이어야 한다.
- `LOGO`는 필수 1개이며 `FileType.LOGO`여야 한다.
- `FIRST_MAP`은 필수 1개이며 `FileType.MAP`이어야 한다.
- `SECOND_MAP`은 선택 0 또는 1개이며 있으면 `FileType.MAP`이어야 한다.
- Trip에서는 `COVER`, `GALLERY` role을 허용하지 않는다.
- `SECOND_MAP`, `nextButtonText`, `previousButtonText`는 모두 있거나 모두 없어야 한다.

## 남은 영향도

### REST 모듈

이번 단계는 Aggregate까지 전환했다. `slcn-rest`는 아직 새 DTO 계약과 aggregate 생성자/메서드 시그니처에 맞춰 추가 수정이 필요할 수 있다.

다음 단계에서 확인할 항목:

- `TripResourceTest`, `TravelResourceTest`의 old request payload 제거
- Travel/Trip request 예제에서 `coverPhotoId`, Trip 직접 파일 ID 제거
- Travel/Trip request payload에 `files` 반영
- Rest layer 컴파일 및 테스트

### 데이터베이스

PostgreSQL `ddl-auto=update`는 삭제된 컬럼/테이블을 자동 제거하지 않는다.

잔존 가능:

- 기존 Travel 하위 테이블
- `trip.logo_file_id`
- `trip.first_map_file_id`
- `trip.second_map_file_id`
- `travels.cover_photo_id`
- `travels.one_line_review`

개발 중 프로젝트이므로 개발 DB 초기화 또는 수동 DDL 정리를 별도 단계에서 수행하면 된다.

### PostgreSQL/MongoDB 일관성

Travel/Trip root는 PostgreSQL, FileBox는 MongoDB에 저장된다. 현재 구현은 분산 트랜잭션을 도입하지 않았다.

현 상태:

- 생성/수정 흐름에서 root 저장 후 FileBox sync를 수행한다.
- FileBox sync 실패 시 예외는 전파된다.
- 생성 실패 보상 삭제는 아직 별도 구현하지 않았다.

추후 필요 시:

- 생성 중 FileBox 저장 실패 시 root 보상 삭제
- 조회 시 root만 있고 FileBox가 없는 데이터에 대한 자동 복구 정책 강화
- Mongo transaction 도입 여부 검토

## 결론

`slcn-aggregate`는 FileBox 설계 기준으로 전환되었다.

- FileBox는 MongoDB document로 저장한다.
- Travel은 root 내부 VO와 FileBox item 조합으로 동작한다.
- Trip은 직접 파일 ID 저장 없이 FileBox role 기반으로 동작한다.
- Aggregate clean test와 Spec+Aggregate 조합 테스트가 통과한다.

다음 단계는 `slcn-rest` 전환과 API 테스트 정리다.
